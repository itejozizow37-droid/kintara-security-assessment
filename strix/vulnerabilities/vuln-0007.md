# Information Disclosure in Service Worker - Internal Architecture and Deployment Details

**ID:** vuln-0007
**Severity:** MEDIUM
**Found:** 2026-07-12 23:05:07 UTC
**Target:** https://kintara.gg
**Endpoint:** /sw.js
**Method:** GET
**CWE:** CWE-611
**CVSS:** 6.5

## Description

The service worker at /sw.js contains detailed comments about the application's caching architecture, deployment process, and internal file structure. The comments reveal that the application is deployed on Render (a cloud platform), uses Three.js from /node_modules/three/, has a maintenance page at /site/maintenance.html, and that the backend uses server.js for routing. The service worker also exposes the caching strategy: /assets/** and /node_modules/three/** are cache-first (which could enable cache poisoning), while /api/** endpoints are never cached. This information helps attackers understand the internal architecture and identify potential attack vectors.

## Impact

The service worker comments expose internal architecture details including: (1) The deployment platform is Render (mentioned in deploy cutover context); (2) The backend routing file is server.js; (3) The maintenance page path is /site/maintenance.html; (4) The cache strategy reveals that /assets/** and /node_modules/three/** are cached first, potentially enabling cache poisoning attacks if an attacker can manipulate these resources; (5) The STATIC_CACHE_VERSION variable controls cache invalidation; (6) The file structure reveals paths like /site/css/, /site/img/, /site/js/ that could be used for further endpoint discovery.

## Technical Analysis

The /sw.js file is the service worker for the Kintara game. It contains extensive comments that describe the caching architecture, deployment platform (Render), internal file structure, and maintenance page location. While service workers are inherently client-side, the comments provide unnecessary internal architecture details that aid reconnaissance. The file also reveals the caching strategy which could be exploited for cache poisoning attacks.

## Proof of Concept

1. Send GET https://kintara.gg/sw.js
2. The server returns the service worker JavaScript file with extensive comments.
3. The comments reveal: deployment on Render, internal file paths, caching strategies, and the maintenance page location.

```
curl -sk --noproxy '*' -H "User-Agent: Mozilla/5.0" "https://kintara.gg/sw.js" | head -50
```

## Remediation

1. Remove all internal architecture comments from the production service worker file.
2. Use a build process to minify the service worker and strip comments before deployment.
3. Do not reference internal deployment platforms (Render) in client-facing code.
4. Do not expose the maintenance page path in the service worker.
5. Ensure the cache version is not predictable or easily guessable.
