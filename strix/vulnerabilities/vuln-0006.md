# Unauthenticated WebSocket Streams All Player Positions, Health, and Status Data

**ID:** vuln-0006
**Severity:** HIGH
**Found:** 2026-07-12 23:04:34 UTC
**Target:** https://kintara.gg
**Endpoint:** /ws/spectate/s{shardId}
**Method:** WSS
**CWE:** CWE-862
**CVSS:** 7.5

## Description

The WebSocket endpoint at wss://kintara.gg/ws/spectate/s{shardId} allows unauthenticated connections and streams real-time data for ALL players on the shard. The "snap" messages contain: player id, name, exact position (x, y, z coordinates), rotation, movement status, current action, outfit/cosmetic data, health points (php), wild shield charges (wsh), combat damage (cv), life epoch (le), and spawn protection (wsp). The connection also receives "online_total" messages with the exact online player count. This data is streamed continuously without any authentication or access control, enabling real-time tracking of every player's location, health, and activities.

## Impact

An unauthenticated attacker can connect to the spectate WebSocket and receive real-time position and status data for every player on any of the 31 game servers. This enables: (1) Real-time player tracking - exact x/z coordinates reveal every player's precise location in the game world, updated continuously; (2) Health and combat monitoring - player HP, shield charges, and combat damage values allow an attacker to identify vulnerable players for targeted attacks; (3) Mass player enumeration - player IDs and names are exposed for all 1,400+ online players; (4) Outfit/cosmetic intelligence - the full outfit schema reveals what cosmetics each player owns; (5) Competitive advantage - knowing enemy positions, health, and equipment provides unfair advantages in PvP combat and boss encounters; (6) The data can be correlated with the chat API, marketplace API, and property API to build complete player dossiers.

## Technical Analysis

The spectate WebSocket at /ws/spectate/s{shardId} is designed for the game's spectate mode, allowing non-players to watch the game. However, it accepts connections without any authentication and streams comprehensive player data. The "snap" message type contains a full snapshot of all players on the shard, including: id (player ID), name (display name), x/y/z (precise 3D coordinates), ry (rotation), mov (movement status), act (current action like mining/fishing/building), pr (region), outfit (full cosmetic schema with hat, top, pants, shoes, skin tone, effects), php (health points 0-100), wsh (wild shield charges), cv (combat damage value), le (life epoch), wsp (spawn protection timer). The "online_total" message type provides the exact number of players online. This data is sent continuously every ~2 seconds, providing a real-time surveillance feed of the entire game.

## Proof of Concept

1. Connect to wss://kintara.gg/ws/spectate/s1 (or any shard 1-31) using a WebSocket client.
2. The server immediately starts sending "snap" messages containing all player data.
3. Each snap message contains an array of player objects with: id, name, x, z, y, ry (position/rotation), mov (moving), act (action), pr (region), outfit (full cosmetic schema), php (health), wsh (shield charges), cv (combat damage), le (life epoch), wsp (spawn protection).
4. The server also sends "online_total" messages with exact player count.
5. No authentication, cookies, or session tokens are required.
6. The connection persists and continues streaming data indefinitely.

```
import asyncio, websockets, json, ssl

async def connect_spectate():
    uri = "wss://kintara.gg/ws/spectate/s1"
    headers = {
        "User-Agent": "Mozilla/5.0 (Windows NT 10.0; Win64; x64) AppleWebKit/537.36 (KHTML, like Gecko) Chrome/126.0.0.0 Safari/537.36",
        "Origin": "https://kintara.gg",
    }
    ssl_ctx = ssl.create_default_context()
    ssl_ctx.check_hostname = False
    ssl_ctx.verify_mode = ssl.CERT_NONE
    
    async with websockets.connect(uri, additional_headers=headers, ssl=ssl_ctx, open_timeout=10) as ws:
        print(f"Connected to {uri}")
        for i in range(5):
            msg = await asyncio.wait_for(ws.recv(), timeout=5)
            data = json.loads(msg)
            if data.get("t") == "snap":
                players = data.get("players", [])
                print(f"Snap: {len(players)} players, online: {data.get('onlineTotal')}")
                if players:
                    print(f"  First player: {json.dumps(players[0], indent=2)}")
            elif data.get("t") == "online_total":
                print(f"Online total: {data.get('n')}")

asyncio.run(connect_spectate())
```

## Remediation

1. Require authentication before accepting WebSocket connections to /ws/spectate/*. Even spectate mode should require a valid session.
2. Reduce the data exposed in spectate snapshots - only send position and basic appearance, not health points, shield charges, combat values, or life epochs.
3. Do not expose player names in spectate snapshots - use anonymized identifiers or only show names for players who have opted in.
4. Implement rate limiting on WebSocket connections to prevent automated mass surveillance.
5. Consider only sending snapshots for players in the immediate vicinity of the spectator's viewport, rather than all players on the entire shard.
6. If spectate mode is intended to be public, limit the refresh rate and data granularity to prevent abuse.
