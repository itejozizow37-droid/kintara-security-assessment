# Unauthenticated Property Owner Data Exposure via /api/property-signs/status

**ID:** vuln-0002
**Severity:** MEDIUM
**Found:** 2026-07-12 22:59:39 UTC
**Target:** https://kintara.gg
**Endpoint:** /api/property-signs/status
**Method:** GET
**CWE:** CWE-862
**CVSS:** 5.3

## Description

The endpoint GET /api/property-signs/status on kintara.gg returns ownership data for all in-game properties (mansions, houses, trailers, and flats) without requiring authentication. The response includes owner names, owner IDs, and lock status for every property. Individual property status can also be queried via /api/mansion/{id}/status. This data was discovered during API reconnaissance by analyzing the client-side game JavaScript bundle.

## Impact

An unauthenticated attacker can enumerate all property owners, their display names, and their numeric player IDs. This data can be cross-referenced with marketplace listings (which expose sellerId and sellerName) and world chat (which exposes playerId and displayName) to build comprehensive player profiles. The lock status also reveals which properties are vulnerable to break-in attempts. The data enables targeted social engineering, in-game stalking, and competitive intelligence about property ownership patterns.

## Technical Analysis

The /api/property-signs/status endpoint serves property display data for the game world. It is accessible without authentication and returns the complete ownership database for all property types. The /api/mansion/{id}/status endpoint also works unauthenticated for IDs 1-3. The data exposed includes ownerName (display name) and ownerId (numeric player ID) for each property, which can be used for player enumeration and cross-referencing with other data sources.

## Proof of Concept

1. Send a GET request to https://kintara.gg/api/property-signs/status without authentication.
2. The server returns HTTP 200 with a JSON body containing all property data organized by type (mansions, houses, trailers, flats).
3. Each property entry includes: ok, sold, ownerName, ownerId, locked.
4. Individual mansion status: GET /api/mansion/1/status returns {"ok":true,"sold":true,"ownerName":"HeadChefX","ownerId":66,"locked":false}
5. The full response includes all 3 mansions, 5 houses, 8 trailers, and 18 flats with their complete ownership data.

```
# Full property data
curl -sk --noproxy '*' \
  -H "User-Agent: Mozilla/5.0 (Windows NT 10.0; Win64; x64) AppleWebKit/537.36 (KHTML, like Gecko) Chrome/126.0.0.0 Safari/537.36" \
  "https://kintara.gg/api/property-signs/status" | python3 -m json.tool

# Individual mansion
curl -sk --noproxy '*' \
  -H "User-Agent: Mozilla/5.0" \
  "https://kintara.gg/api/mansion/1/status"
```

## Remediation

1. Require authentication before returning property ownership data.
2. Minimize the data exposed - only return whether a property is sold/available, not the owner's name and ID.
3. Remove the lock status from unauthenticated responses, as it reveals security-relevant information about properties.
4. If spectate mode needs property data, provide only the property type and sold status, not owner identity.
