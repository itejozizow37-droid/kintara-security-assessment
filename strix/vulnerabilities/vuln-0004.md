# adminBypass Flag Exposure in /api/servers Response

**ID:** vuln-0004
**Severity:** MEDIUM
**Found:** 2026-07-12 23:00:36 UTC
**Target:** https://kintara.gg
**Endpoint:** /api/servers
**Method:** GET
**CWE:** CWE-862
**CVSS:** 5.3

## Description

The endpoint GET /api/servers on kintara.gg returns the complete server list including an "adminBypass" flag. This flag, which appears to control whether admin authentication is bypassed on specific servers, is exposed to all unauthenticated users. The response includes 31 game servers with details about population, capacity, membership requirements, and the adminBypass flag. The flag is currently set to false, but its exposure reveals the existence of an admin bypass mechanism that could be targeted.

## Impact

The adminBypass flag in the server response reveals the existence of an admin authentication bypass mechanism. While currently set to false, this information disclosure helps an attacker understand the application's access control architecture and provides a specific target for auth bypass attacks. If this flag could be manipulated (via parameter injection, header manipulation, or server-side state modification), it could grant full admin access to the game's 60+ admin endpoints, including player banning, gold gifting, item granting, and server control.

## Technical Analysis

The /api/servers endpoint returns server metadata for the game's load balancer/lobby system. The response includes an "adminBypass" boolean field at the root level alongside the servers array. This flag appears to be a server configuration value that controls whether admin authentication is enforced. Its presence in the API response suggests the server has a mechanism to disable admin auth checks, and exposing this flag to unauthenticated users reveals the internal access control architecture. The client-side game code (game.js) contains checks like `e.isAdmin===true` which sets `sn` (localPlayerIsAdmin), and admin users bypass the $KINS token requirement entirely.

## Proof of Concept

1. Send GET https://kintara.gg/api/servers without authentication.
2. The response returns 31 servers with various metadata.
3. The response includes "adminBypass":false at the root level of the JSON response.
4. The full response structure: {"ok":true,"servers":[{...31 servers...}],"adminBypass":false}
5. Servers 1-5 are "Kintara Club" servers requiring membership, servers 6-31 are regular servers.

```
curl -sk --noproxy '*' \
  -H "User-Agent: Mozilla/5.0 (Windows NT 10.0; Win64; x64) AppleWebKit/537.36 (KHTML, like Gecko) Chrome/126.0.0.0 Safari/537.36" \
  "https://kintara.gg/api/servers" | python3 -c "import json,sys; d=json.load(sys.stdin); print(f'adminBypass: {d.get(\"adminBypass\")}'); print(f'Servers: {len(d.get(\"servers\",[]))}')"
```

## Remediation

1. Remove the adminBypass flag from the API response - this is internal server configuration that should not be exposed to clients.
2. Ensure that admin access control is enforced server-side and cannot be bypassed by any client-side flag or parameter.
3. Audit all admin endpoints to verify they properly validate server-side admin status, not relying on any client-supplied flag.
4. If the adminBypass mechanism is needed for development, restrict it to a dev/staging environment and never expose it in production API responses.
