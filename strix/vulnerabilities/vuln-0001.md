# Unauthenticated World Chat Exposure via /api/spectate/chat/bootstrap

**ID:** vuln-0001
**Severity:** MEDIUM
**Found:** 2026-07-12 22:59:17 UTC
**Target:** https://kintara.gg
**Endpoint:** /api/spectate/chat/bootstrap
**Method:** GET
**CWE:** CWE-862
**CVSS:** 5.3

## Description

The endpoint GET /api/spectate/chat/bootstrap on kintara.gg returns approximately 100 recent world chat messages without requiring any authentication. Each message exposes sensitive player data including: playerId, displayName, shardId, worldX/worldZ (precise player coordinates), message text, walletPubkey, and timestamp. This was discovered during reconnaissance of the API surface by analyzing the client-side JavaScript bundle and testing endpoints.

## Impact

An unauthenticated attacker can monitor all player communications and locations in real-time. This enables: (1) Player tracking - coordinates reveal exact positions of players in the game world, (2) Identity correlation - displayNames and playerIds can be cross-referenced with marketplace listings and property records to build complete player profiles, (3) Social engineering - chat messages reveal player activities, trades, and relationships that can be exploited for targeted attacks, (4) Competitive intelligence - chat reveals in-game activities like boss runs, PvP plans, and market trading.

## Technical Analysis

The /api/spectate/chat/bootstrap endpoint is designed for the spectate mode of the game (allowing non-players to watch). However, it exposes full chat history with detailed player metadata without any authentication or access control. The response includes an array of message objects, each containing playerId, displayName, shardId, worldX, worldZ (player coordinates), message text, walletPubkey, and createdAtMs timestamp. The spectate chat endpoint at /api/spectate/chat also returns similar data. The fanout subdomain (fanout.kintara.gg) also serves this endpoint, expanding the attack surface.

## Proof of Concept

1. Send a GET request to https://kintara.gg/api/spectate/chat/bootstrap without any authentication headers or cookies.
2. The server returns HTTP 200 with a JSON body containing an array of chat messages.
3. Each message contains: id, playerId, shardId, displayName, walletPubkey, region, worldX, worldZ, message, createdAtMs.
4. Example response (truncated): {"ok":true,"maxId":460609,"shardId":1,"messages":[{"id":459014,"playerId":3809,"shardId":1,"displayName":"ayegrimes","walletPubkey":"","region":"world","worldX":-2.5,"worldZ":-2.5,"message":"rigged","createdAtMs":1783883595132},...]}

## Remediation

1. Require authentication (session cookie) before returning chat messages, even for spectate mode.
2. Reduce the amount of PII exposed in chat messages - remove or hash playerId and walletPubkey fields for unauthenticated viewers.
3. Do not expose precise player coordinates (worldX, worldZ) in the spectate chat API; these should only be available to authenticated players in the same shard.
4. Implement rate limiting on the endpoint to prevent automated enumeration.
5. Consider only returning the most recent N messages (e.g., 10) for spectate viewers, rather than the full history.
